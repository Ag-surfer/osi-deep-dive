## Bootstrapping a host onto the network

A device that just booted knows nothing: no [IP address](../../network/), no subnet mask, no default
gateway, no [DNS](../dns/) server. **DHCP** automates that configuration so a laptop or phone joins any
network without manual setup. It's an [Application-layer](../../application/) protocol with a chicken-and-
egg twist — the client needs to talk to a server before it has an IP — solved with **broadcast**.

## The DORA exchange

DHCP configures a host in four messages, mnemonically **DORA**. Because the client has no address yet, it
sends from `0.0.0.0` to the broadcast address `255.255.255.255`:

<Callout variant="insight" title="Why Request is broadcast, and what's in the Ack">
Several DHCP servers may answer a Discover. The client picks one offer but **broadcasts** its Request, so
the *other* servers hear that their offer wasn't taken and free the address they had reserved. The final
**Ack** delivers far more than an IP: the **subnet mask**, **default gateway**, **DNS servers**, lease
time, and dozens of optional parameters (domain name, NTP servers, PXE boot file, and vendor options).
DHCP is how almost every networked device learns *everything* it needs to participate.
</Callout>

## Leases and renewal

An address is a **lease**, not a gift — granted for a fixed time so the pool recycles as devices come and
go. The client renews automatically: at **50% of the lease (T1)** it unicasts a renewal to its server;
if that fails, at **87.5% (T2)** it broadcasts to *any* server (rebinding); if the lease fully expires it
must start over. A clean shutdown can send **DHCPRELEASE** to return the address early.

## Relay agents: DHCP across subnets

DHCP's discovery is **broadcast**, and routers don't forward broadcasts — so a single central DHCP server
couldn't serve other subnets. A **relay agent** (a router with an `ip helper-address`) catches the
broadcast and **unicasts** it to the central server, tagging it with which subnet it came from so the
server picks the right pool. This is what lets one DHCP server configure an entire campus.

<Callout variant="note" title="The IPv6 alternative: SLAAC">
IPv6 can configure addresses **without** a DHCP server at all, via **SLAAC** (Stateless Address
Autoconfiguration): the router advertises a prefix, and the host forms its own address from it. **DHCPv6**
still exists for handing out DNS servers and other options, often *alongside* SLAAC — a different
division of labor than IPv4's all-in-one DHCP.
</Callout>

## Failure modes & troubleshooting

<Callout variant="warning" title="Symptoms and the cause behind them">
- **Host self-assigns a 169.254.x.x address (APIPA)** — it got no DHCP response at all: no server
  reachable, a dead relay/`ip helper`, or a broken link.
- **Intermittent 'no address available'** — the **pool is exhausted** (too small for the device count, or
  leases too long with high churn); enlarge the pool or shorten leases.
- **Duplicate IP / address conflicts** — a static IP overlapping the DHCP range, or two servers with
  overlapping pools.
- **Wrong subnet's address handed out** — relay/Option 82 misconfiguration so the server picks the wrong
  pool.
</Callout>

## Security considerations

<Callout variant="security" title="An unauthenticated broadcast protocol">
- **Rogue DHCP server** — anyone on the LAN can answer a Discover, handing out a malicious gateway/DNS to
  become a man-in-the-middle. Defense: **DHCP snooping** marks only designated ports as trusted and drops
  server messages from the rest.
- **DHCP starvation** — flooding Discovers with spoofed MACs to drain the pool (a DoS); **port security**
  and snooping rate-limits defend it.
- **No authentication** — DHCP has essentially none, which is exactly why the switch-level protections
  (snooping, port security, 802.1X before granting access) carry the load.
</Callout>
