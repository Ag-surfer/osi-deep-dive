## Connecting processes, not just hosts

The Network layer gets a packet to the right **host**. But a host runs dozens of programs at once — a
browser, a mail client, SSH. The Transport layer delivers data to the right **process** using **port
numbers**, and chooses the *contract*: TCP's reliable ordered stream, or UDP's bare datagram. This is
where the **end-to-end principle** pays off — reliability lives at the edges, so the network core stays
simple.

## Ports and multiplexing

A port is a 16-bit number (0–65535). A connection is identified by the **4-tuple**
`(source IP, source port, dest IP, dest port)` — which is how your machine keeps hundreds of
simultaneous connections straight. Ports below 1024 are **well-known** (80 = HTTP, 443 = HTTPS,
22 = SSH, 53 = DNS); clients pick high **ephemeral** ports.

## TCP: the reliable ordered stream

TCP turns IP's lossy, unordered, best-effort packets into a **reliable, in-order byte stream**. It adds
connection setup, sequence numbers, acknowledgments and retransmission, **flow control**, and
**congestion control**.

### Flow control vs. congestion control

Two different problems, often confused:

- **Flow control** protects the *receiver* from being overrun — the advertised **Window Size** says
  "send at most this much more." A sliding window.
- **Congestion control** protects the *network* from overload — TCP maintains its own *congestion
  window* and ramps carefully: **slow start** (exponential growth) until a threshold, then **congestion
  avoidance** (linear growth), halving on loss (**AIMD** — additive increase, multiplicative decrease).
  **Fast retransmit/recovery** reacts to three duplicate ACKs without waiting for a timeout.

<SlidingWindow />

<WorkedExample title="Window needed to fill a fast, long link">
Throughput on TCP is bounded by `window / RTT`. To saturate a **100 Mbps** path with an **80 ms** RTT:

1. Bandwidth-delay product (BDP) = `100×10⁶ bits/s × 0.080 s = 8×10⁶ bits = 1 MB`.
2. So the sender needs ~**1 MB of data in flight** to keep the pipe full.
3. The original TCP window field maxes at 65,535 bytes — far too small. The **window scaling** option
   (negotiated in the handshake) multiplies it so modern stacks can reach the needed 1 MB.

This is why a high-bandwidth, high-latency link ("long fat network") performs poorly without window
scaling: a tiny window throttles throughput regardless of available bandwidth.
</WorkedExample>

## UDP: thin and fast

<RFCRef n={768}>UDP</RFCRef> adds almost nothing to IP — just ports and a checksum — in an **8-byte**
header. No handshake, no ordering, no retransmission. That's a feature for DNS lookups, real-time
audio/video, and gaming, where a late packet is worse than a lost one.

## QUIC: transport, reinvented in user space

**QUIC** (<RFCRef n={9000} />, the foundation of HTTP/3) is what you get when engineers redesign the
Transport layer with thirty years of TCP lessons — and then have to *ship* it on an internet whose
middleboxes only pass TCP and UDP. The answer: build it **on top of UDP**, in user space.

| Problem with TCP | QUIC's answer |
| --- | --- |
| Separate TCP + TLS handshakes (2–3 RTTs) | Combined transport + TLS 1.3 setup (1 RTT, 0-RTT on resume) |
| One lost packet stalls *all* data behind it (head-of-line blocking) | Independent **streams** — loss in one stream never blocks another |
| A connection is pinned to one IP/port 4-tuple | **Connection IDs** — survive switching from Wi-Fi to cellular mid-download |
| Upgrading TCP means upgrading every OS kernel and middlebox | User-space implementation — browsers ship improvements in weeks |

The head-of-line fix matters most: a web page loads dozens of resources, and with HTTP/2-over-TCP a
single lost packet freezes *all* of them, because TCP guarantees one strictly ordered byte stream.
QUIC's per-stream ordering means one unlucky image stalls only itself.

<Callout variant="insight" title="Ossification: why QUIC hides from the network">
Decades of firewalls and NATs grew opinions about what TCP "should" look like, making the TCP header
effectively unchangeable — the internet's protocols **ossified**. QUIC encrypts almost its entire
header so middleboxes *can't* inspect or meddle with it, deliberately keeping tomorrow's transport
evolvable. A protocol designed so the network can't see it: that's a profound lesson about where
flexibility survives in a layered system.
</Callout>

## Failure modes & troubleshooting

<Callout variant="warning" title="Symptoms at Layer 4">
- **Ephemeral port exhaustion** — a busy proxy runs out of source ports for new connections; you see
  connection failures under load.
- **TIME_WAIT pileup** — many short connections leave sockets lingering; tune reuse rather than disabling.
- **Throughput capped far below link rate** — often a small window on a high-BDP path (enable window
  scaling), or persistent loss forcing congestion-window collapse.
- **Connection resets (RST)** — the peer refused or aborted; distinguish from silent drops (firewall).
</Callout>

## Security considerations

<Callout variant="security" title="Attack surface at Layer 4">
- **SYN flood** — send SYNs, never complete the handshake, exhausting the server's half-open
  connection table (DoS). Defense: **SYN cookies**.
- **TCP RST/sequence injection** — forging a RST or data with a guessed sequence number to hijack or
  tear down a session. Defense: randomized initial sequence numbers, and encryption (TLS) above.
- **Port scanning** — probing which ports are open to map services. Defense: firewalls, rate limiting.
- **Off-path attacks** — mitigated by the 4-tuple + random ports/ISNs raising the guessing difficulty.
</Callout>
