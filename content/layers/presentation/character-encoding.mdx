## Bytes are not text

A file or packet is bytes; text is an interpretation of those bytes. The [Presentation
layer](../../presentation/) owns that interpretation — the **character encoding** that maps between
human characters and the bytes on the wire. Get sender and receiver to agree and text just works; let
them disagree and you get **mojibake** — "café" rendered as "cafÃ©". Two concepts must stay separate:

- A **code point** is an abstract number assigned to a character (Unicode `U+0041` = "A", `U+1F600` =
  "😀"). It says *which* character, not how it's stored.
- An **encoding** maps code points to actual bytes (UTF-8, UTF-16). It says *how* the character is
  stored.

## From ASCII to Unicode

**ASCII** (1963) used 7 bits for 128 characters — enough for English, and the reason byte 65 is "A"
everywhere. But the world has far more than 128 characters, and the 1990s sprawl of incompatible 8-bit
"code pages" (Latin-1, Windows-1252, Shift-JIS…) meant the same byte meant different characters in
different locales — the original mojibake. **Unicode** fixed this with **one universal code space**
(~1.1 million code points, ~150k assigned) covering every script, symbol, and emoji. Now there's one
answer to "which character is this?" — leaving only the question of how to encode those code points.

## UTF-8: the encoding that won

**UTF-8** encodes each code point in **1 to 4 bytes**, and its design is a small masterpiece:

| Code point range | Bytes | Pattern |
| --- | --- | --- |
| U+0000 – U+007F | 1 | `0xxxxxxx` |
| U+0080 – U+07FF | 2 | `110xxxxx 10xxxxxx` |
| U+0800 – U+FFFF | 3 | `1110xxxx 10xxxxxx 10xxxxxx` |
| U+10000 – U+10FFFF | 4 | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` |

<Callout variant="insight" title="Why UTF-8 took over the internet">
- **ASCII-compatible** — any 7-bit ASCII text is *already* valid UTF-8, byte-for-byte. Decades of
  English text and protocols (HTTP headers, code) kept working unchanged.
- **No NUL bytes mid-character** and no embedded ASCII bytes — a UTF-8 "é" never contains a byte that
  looks like `/` or NUL, so C string functions and path parsers don't choke.
- **Self-synchronizing** — the high bits reveal each byte's role: `0…` = single byte, `11…` = a leader
  (with the count of following bytes), `10…` = a continuation. Land anywhere in a stream and you can
  find the next character boundary, and a single corrupted byte resyncs instead of cascading.
- **Endian-free** — being byte-oriented, it has no byte-order ambiguity (unlike UTF-16).
These properties are why UTF-8 is now ~98% of the web and the correct default for essentially everything.
</Callout>

## UTF-16, surrogates, and the BOM

**UTF-16** (used internally by Windows, Java, JavaScript strings) encodes most characters in 2 bytes but
needs a **surrogate pair** (two 16-bit units) for code points above U+FFFF — which is why an emoji can
count as "length 2" in JavaScript. Being multi-byte, UTF-16 also needs a **Byte Order Mark** (BOM) to
signal big- vs little-endian — a complication UTF-8 simply doesn't have.

<Callout variant="note" title="Normalization: same text, different bytes">
Unicode often offers **two ways to write the same character**: "é" can be the single code point U+00E9,
or "e" (U+0065) followed by a combining accent (U+0301). They look identical but are different byte
sequences, so a naive `==` or a password/username comparison can fail. **Normalization** (NFC composes,
NFD decomposes) rewrites text into a canonical form so equal-looking strings compare equal — essential
for search, identifiers, and security checks.
</Callout>

## Failure modes & troubleshooting

<Callout variant="warning" title="Symptoms rooted in encoding">
- **Mojibake ("café" → "cafÃ©")** — UTF-8 bytes decoded as Latin-1/Windows-1252 (or vice versa); fix the
  declared charset, not the data.
- **"" question-marks / boxes** — the encoding is right but the **font** lacks that glyph, or data was
  lossily transcoded through a smaller charset.
- **Emoji/string length surprises** — counting UTF-16 code units or bytes instead of code points (or
  grapheme clusters); a "👍🏽" can be several code points.
- **Username/password mismatch that looks identical** — unnormalized Unicode; normalize (NFC) before
  comparing or hashing.
- **A leading "" in the first cell of a CSV** — a UTF-8 BOM the consumer didn't strip.
</Callout>
