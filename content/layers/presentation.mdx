## The translator between machine and wire

Two machines may represent the same information completely differently — different character sets,
different byte orders, different struct layouts. The Presentation layer is the **translator**: it
converts between an application's internal data and a common representation suitable for transmission.
Its three jobs are **encoding/translation**, **compression**, and **encryption**.

## Character encoding

Text is just numbers, and the mapping matters. **ASCII** (7 bits) covers English; **Unicode** assigns a
*code point* to every character in every script, and **UTF-8** encodes those code points as 1–4 bytes —
backward-compatible with ASCII and now dominant on the web. Get the encoding wrong and you get
**mojibake** (`café` → `cafÃ©`): the bytes were fine, the *interpretation* was not.

<WorkedExample title="How UTF-8 encodes 'é' (U+00E9)">
The code point `U+00E9` = 233 needs more than 7 bits, so UTF-8 uses a **2-byte** form:

1. Template for 2-byte sequences: `110xxxxx 10xxxxxx` (11 payload bits).
2. `0xE9` in binary is `000 1110 1001` → pad to 11 bits: `00011 101001`.
3. Fill the template: `11000011 10101001` = bytes `0xC3 0xA9`.

So `é` travels as two bytes, `C3 A9`. An ASCII-only reader sees two "garbage" bytes — the classic
encoding-mismatch bug.
</WorkedExample>

## Serialization

In-memory objects (graphs of pointers) can't be sent as-is; they must be **serialized** into a byte
stream and **deserialized** on the other side. The format is a Presentation-layer choice with real
trade-offs:

| Format | Shape | Trade-off |
| --- | --- | --- |
| **JSON** | Text, human-readable | Ubiquitous, verbose, no schema by default |
| **XML** | Text, verbose, schema-rich | Powerful but heavy |
| **Protocol Buffers** | Binary, schema-first | Compact and fast; needs the schema to read |
| **ASN.1 (DER)** | Binary, formally specified | Used in X.509 certificates, LDAP, telecom |

## Byte order: the oldest translation problem

Multi-byte numbers have two natural layouts: **big-endian** (most significant byte first — how humans
write numbers) and **little-endian** (least significant byte first — how x86 and most ARM systems
store them). The 32-bit number `0x0A0B0C0D` is four bytes — but *which order do they travel in?*

<WorkedExample title="The same number, two machines, one wire">
1. A little-endian x86 sender holds `0x0A0B0C0D` in memory as `0D 0C 0B 0A`.
2. Network protocols define **network byte order = big-endian** (set down alongside
   <RFCRef n={791}>IPv4</RFCRef> itself), so the sender must transmit `0A 0B 0C 0D`.
3. Every numeric header field — ports, lengths, sequence numbers — goes through `htons()`/`htonl()`
   ("host to network short/long") on the way out and `ntohs()`/`ntohl()` on the way in.

Skip the conversion and port `80` (`0x0050`) is read as port `20480` (`0x5000`) — a bug every
network programmer writes exactly once.
</WorkedExample>

This is the Presentation layer's purest case: both machines hold the *same value*; only its
**representation** differs, and somebody must own the translation. Serialization formats make the
same decision for you — JSON sidesteps it with text; Protocol Buffers specifies little-endian and
shields you from it.

## Compression and encryption

- **Compression** shrinks data before it hits the wire — lossless (gzip, Brotli for HTTP; PNG) or
  lossy (JPEG, H.264, where discarding imperceptible detail buys huge savings).
- **Encryption** transforms data so only the intended party can read it. **TLS** is the towering
  example: it negotiates ciphers and keys, then encrypts the application's bytes — confidentiality,
  integrity, and authentication, sitting exactly where the Presentation layer says encryption belongs.

<Callout variant="insight" title="Why TLS is 'presentation'">
TLS doesn't care whether it's wrapping HTTP, SMTP, or IMAP — it transforms the *representation* of
whatever bytes pass through (encrypting them) without understanding their meaning. That representation
transformation is the essence of Layer 6, even though TLS spans session and presentation concerns.
</Callout>

The negotiation itself is worth seeing. In **TLS 1.3** a single round trip agrees on cipher suites,
exchanges keys, and authenticates the server before a byte of application data flows:

When this handshake fails — an expired certificate, no cipher in common, a hostname mismatch — the
TCP connection is alive and the application never gets a chance to run: a failure that lives
precisely *between* Layers 4 and 7.

## Failure modes & troubleshooting

<Callout variant="warning" title="Symptoms at the presentation level">
- **Mojibake / `?` boxes** — encoding mismatch (declared charset ≠ actual bytes).
- **Deserialization errors** — schema drift between sender and receiver, or untrusted input parsed
  unsafely.
- **Double-compression / wrong content-encoding** — gzip headers mismatched, producing garbage.
- **Certificate/cipher mismatch** — TLS negotiation fails when client and server share no acceptable
  cipher suite or the cert chain is broken.
</Callout>

## Security considerations

<Callout variant="security" title="Attack surface at Layer 6">
- **Insecure deserialization** — parsing untrusted serialized data can execute attacker-controlled
  code (a top web vulnerability). Defense: validate, use safe formats, never deserialize untrusted
  input into live objects.
- **Weak/legacy crypto** — old TLS versions and ciphers (SSLv3, RC4) are broken; enforce TLS 1.2+.
- **Padding-oracle & compression attacks** (CRIME/BREACH) — combining compression with encryption can
  leak secrets; mitigated by disabling TLS-level compression and separating secrets from attacker input.
</Callout>
