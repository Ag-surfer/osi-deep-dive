## Why compression is possible

The [Presentation layer](../../presentation/) prepares the application's data for the wire, and one of
its biggest jobs is making that data **smaller**. Compression works because real data is **redundant** —
some symbols are far more common than others, and patterns repeat. Information theory puts a floor on it:
a source's **entropy** is the average number of bits genuinely needed per symbol. Compression's goal is
to spend bits in proportion to *surprise* — short codes for the predictable, long codes for the rare —
squeezing the representation down toward that entropy floor. Two big families do this: **entropy coding**
(optimize the code per symbol) and **dictionary coding** (replace repeated strings with references).

## Entropy coding: Huffman

**Huffman coding** builds the optimal **prefix-free** code for a known set of symbol frequencies: no
code is a prefix of another (so the stream decodes unambiguously with no delimiters), and the most
frequent symbols get the shortest codes. It builds the tree bottom-up, repeatedly merging the two
lowest-frequency nodes. Step through it:

<Callout variant="insight" title="Why the prefix-free property matters">
Because no code is a prefix of another, a decoder can read bits and emit a symbol the instant it matches
a leaf — no separators, no ambiguity. And because the tree places frequent symbols shallow, the
**average** code length drops below the fixed-length `⌈log₂(alphabet)⌉` bits. Here, ~2.24 bits/symbol vs.
3 for a fixed code — and the bigger the frequency skew, the bigger the win. **Arithmetic / range coding**
(and modern **ANS**) push even closer to the entropy limit by coding fractions of a bit per symbol.
</Callout>

## Dictionary coding: LZ77 and friends

Huffman exploits symbol *frequency*; it can't capture that the word "the" repeats a thousand times. The
**Lempel–Ziv** family does. **LZ77** slides a window over the data and replaces a repeated string with a
**(distance, length)** back-reference to its earlier occurrence — "go back 80 bytes and copy 12." LZ78/
**LZW** instead build an explicit dictionary of seen strings (LZW famously powered GIF and `compress`).
Dictionary coding turns repetition into tiny pointers.

<Callout variant="note" title="DEFLATE: the workhorse — both at once">
The most-deployed lossless format, **DEFLATE** (<RFCRef n={1951} />), combines them: **LZ77** first
replaces repeats with back-references, then **Huffman** entropy-codes the resulting symbols and
distances. That two-stage pipeline is what powers **gzip**, **zlib**, **PNG**, and the classic HTTP
`Content-Encoding: gzip`. Modern successors trade more CPU/memory for better ratios: **Brotli**
(<RFCRef n={7932} />, with a built-in dictionary of common web strings) for HTTP, and **Zstandard
(zstd)** for fast general-purpose compression — both now common as HTTP content encodings.
</Callout>

## Lossless vs. lossy

Everything above is **lossless** — decompression reproduces the input bit-for-bit, mandatory for text,
code, and archives. **Lossy** compression instead *discards information the human senses barely notice*,
achieving far smaller files for media:

- **JPEG** transforms image blocks with the **Discrete Cosine Transform**, then quantizes away
  high-frequency detail the eye is insensitive to (then Huffman-codes the rest).
- **H.264/H.265/AV1** add motion compensation between video frames; **MP3/AAC/Opus** drop sounds masked
  by louder ones (psychoacoustics).

The dial is **rate vs. distortion**: spend more bits for higher fidelity, fewer for a smaller file. The
choice between lossless and lossy is a data-type decision the Presentation layer makes for the
application.

## Failure modes & troubleshooting

<Callout variant="warning" title="Pitfalls of compression">
- **Already-compressed data won't shrink** — re-gzipping a JPEG or a zip wastes CPU for ~0% gain;
  there's no redundancy left to exploit.
- **CRIME / BREACH attacks** — compressing **secret + attacker-controlled** data together leaks the
  secret through the compressed *size*; never compress mixed-trust content under TLS without mitigation.
- **Decompression bombs** — a tiny archive that expands to gigabytes (a DoS); bound the output size.
- **CPU vs. ratio** — Brotli/zstd at max level can cost more time than they save on a fast link;
  match the level to the use (precompress static assets, go light on dynamic ones).
</Callout>
