## Where the network meets the user

The Application layer is the one people actually experience — though not the apps themselves, but the
**protocols** applications use to talk over the network. It defines message formats and exchange rules
for concrete services: the web (**HTTP**), name resolution (**DNS**), email (**SMTP/IMAP**), secure
shells (**SSH**), address assignment (**DHCP**), file transfer (**FTP**). Everything below exists to
carry these conversations.

## Anatomy of an HTTP exchange

HTTP is delightfully readable: a request is a method + path + headers; a response is a status code +
headers + body.

```http
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1256

<!doctype html> …
```

It is **stateless** by design — each request stands alone, which is why cookies and tokens (Session-
layer concerns) exist to thread identity across requests. **HTTPS** is simply HTTP carried inside a TLS
tunnel (Presentation-layer encryption), and **HTTP/3** runs over QUIC.

### HTTP grew up: 1.0 → 1.1 → 2 → 3

Each HTTP version attacks the previous one's performance ceiling — a thirty-year case study in
protocol evolution:

| Version | Key change | The problem it solved |
| --- | --- | --- |
| **1.0** (1996) | One TCP connection **per request** | — (it created the problem) |
| **1.1** (1997) | **Keep-alive** connections; chunked transfer; `Host` header | Handshake-per-request waste; virtual hosting on shared IPs |
| **2** (2015) | **Binary framing**: many streams multiplexed on one connection; header compression (HPACK) | Browsers opening 6 parallel connections and still serializing requests |
| **3** (2022) | Same semantics over **QUIC** instead of TCP | One lost packet stalling *every* HTTP/2 stream (transport-level head-of-line blocking) |

The head-of-line story is the connecting thread — and a great interview answer. HTTP/1.1 had
*application-level* HOL blocking (responses must return in order, so one slow response blocks the
connection). HTTP/2 fixed that with multiplexed streams, only to expose *transport-level* HOL
blocking: TCP guarantees one ordered byte stream, so a single lost segment halts every stream riding
it. HTTP/3 moves to QUIC, whose independent streams finally make one resource's bad luck private.
Same `GET`/`200 OK` semantics for thirty years; everything underneath rebuilt twice.

## DNS: the lookup behind every connection

Before your browser can open a TCP connection, it must turn `example.com` into an IP address. DNS does
this with a compact 12-byte header.

A recursive resolver walks the DNS hierarchy from the root down:

<WorkedExample title="Why the first byte of a web page takes so long">
Loading `https://example.com` cold (nothing cached) costs a *round trip* (RTT) at each layer's setup —
this is why latency, not bandwidth, dominates page-load feel:

1. **DNS** lookup — ~1 RTT (often cached, sometimes more).
2. **TCP** three-way handshake — 1 RTT (Layer 4).
3. **TLS 1.3** handshake — 1 RTT (Layers 5–6).
4. **HTTP** request → first response byte — 1 RTT (Layer 7).

That's **~4 RTTs** before content arrives. On a 100 ms-RTT mobile link, ~400 ms of pure waiting,
regardless of how fast the pipe is. **HTTP/3 + QUIC** fold transport and TLS setup together (and 0-RTT
resumption can skip the handshake entirely), cutting this dramatically — a direct payoff of
understanding what each layer adds.
</WorkedExample>

## Email: one message, three protocols

Email looks like one service but is a relay race between asymmetric protocols — push to deliver,
pull to read:

<SequenceDiagram
  actors={["Your client", "Your mail server", "Their mail server", "Recipient"]}
  steps={[
    { from: 0, to: 1, label: "SMTP submit (port 587, authenticated)" },
    { from: 1, to: 2, label: "SMTP relay (port 25 — found via DNS MX record)" },
    { from: 2, to: 3, label: "IMAP fetch (port 993) — when they check mail", dashed: true },
  ]}
  caption="SMTP pushes mail toward the destination server; IMAP lets the recipient pull it on their schedule. The handoff point is a DNS MX lookup."
/>

- **SMTP** (<RFCRef n={5321} />) is push-only and text-based — and predates spam: it has no built-in
  sender authentication, which is why **SPF**, **DKIM** (cryptographic signatures), and **DMARC**
  were bolted on decades later via DNS records.
- **IMAP** keeps mail on the server and synchronizes state across devices (read flags, folders);
  legacy **POP3** simply downloads and deletes — fine in 1996, useless with three devices.
- **MIME** solves a Presentation-layer problem inside an Application protocol: SMTP moves 7-bit
  text, so attachments are Base64-encoded into labeled body parts (`Content-Type: image/png`) —
  which is why attachments inflate ~33% in transit.

The MX-record handoff is the interview detail: the sending server asks DNS not for the domain's
address, but *"who accepts mail for this domain?"* — infrastructure indirection that let
organizations outsource mail decades before "the cloud" had a name.

## DHCP: how a host gets on the network at all

Before any of the above can happen, a freshly connected device knows *nothing* — no IP address, no
gateway, no DNS server. **DHCP** (<RFCRef n={2131} />) bootstraps it with a four-step exchange
nicknamed **DORA**, carried over UDP broadcast because the client doesn't yet have an address to be
reached at:

<SequenceDiagram
  actors={["New host", "DHCP server"]}
  steps={[
    { from: 0, to: 1, label: "DISCOVER — broadcast: any DHCP servers out there?" },
    { from: 1, to: 0, label: "OFFER — here's 192.168.1.23, lease 24 h", dashed: true },
    { from: 0, to: 1, label: "REQUEST — I'd like that one, please (broadcast)" },
    { from: 1, to: 0, label: "ACK — it's yours: gateway, mask, DNS included", dashed: true },
  ]}
  caption="DHCP's DORA exchange: a host with no address uses broadcast to acquire one, plus everything it needs to use the network."
/>

The ACK delivers a complete starter kit: address, subnet mask, default gateway, DNS servers, and a
**lease** the host must renew. It's a beautiful illustration of layering's chicken-and-egg edge case —
an Application-layer protocol whose whole job is configuring Layer 3, running before Layer 3 is
configured, possible only because broadcast frames need no IP routing.

<Callout variant="security" title="Rogue DHCP">
Nothing authenticates a DHCP OFFER. A malicious server on the LAN that answers faster can hand out
*itself* as gateway and DNS — silently routing every victim's traffic through an attacker box.
Defense: **DHCP snooping** on switches, which permits OFFERs only from trusted ports. Note the layer
interplay: a Layer-7 protocol attacked at Layer 2, defended at Layer 2.
</Callout>

## Architectural models

- **Client–server** — the dominant model (browser↔web server, mail client↔mail server). Simple,
  centralized, easy to secure and scale with load balancers.
- **Peer-to-peer** — every node is both client and server (BitTorrent, blockchains). No central point,
  resilient, but harder to coordinate and secure.

## Failure modes & troubleshooting

<Callout variant="warning" title="Symptoms at Layer 7">
- **NXDOMAIN / resolution failures** — the name doesn't exist or DNS is misconfigured; `dig`/`nslookup`
  isolate it from connectivity.
- **4xx vs 5xx** — 4xx is the client's fault (bad request, unauthorized, not found); 5xx is the
  server's (it received the request but failed).
- **Slow first byte, fast transfer** — latency/RTT-bound setup (DNS+TCP+TLS), not bandwidth. Caching,
  CDNs, and HTTP/3 attack this.
- **Mixed content / cert errors** — HTTPS page loading HTTP resources, or an invalid certificate chain.
</Callout>

## Security considerations

<Callout variant="security" title="Attack surface at Layer 7">
- **Injection (SQLi, command, XSS)** — untrusted input interpreted as code/markup. The #1 web risk;
  defense is rigorous input validation and parameterized queries/output encoding.
- **DNS cache poisoning** — forging responses to redirect victims; defense is **DNSSEC** (signed
  records) and randomized query IDs/ports.
- **Phishing & credential theft** — social-layer attacks; defense is MFA, user education, and email
  authentication (SPF/DKIM/DMARC).
- **Application-layer DDoS** — floods of "expensive" valid requests; defense is rate limiting, WAFs,
  and CDNs.
</Callout>
