## Two ideas, half the internet's bytes

Almost every byte you receive is compressed — gzip'd HTML, Brotli'd JavaScript, PNG images, ZIP
downloads — and almost all of it stands on **two ideas from the 1950s and 1970s** stacked
together. Understanding them demystifies the whole field and explains a surprising security
hazard.

<ProtocolDiagram id="compression" />

## Idea 1 — Huffman coding (entropy)

Fixed-width encoding wastes space: ASCII spends 8 bits on a rare `z` and 8 on a ubiquitous space.
**Huffman coding** (1952, a grad-school term paper) assigns *short* bit-codes to frequent symbols
and *long* ones to rare symbols, with a clever guarantee — no code is a prefix of another, so the
stream decodes unambiguously with no delimiters. The size it converges toward is the data's
**entropy**, Shannon's information-theoretic floor — the same Shannon, the same 1948 paper, that
bounds [what a channel can carry](/layers/physical/). Compression and transmission turn out to be
the same theory viewed from two sides.

## Idea 2 — LZ77 (dictionaries)

Real data isn't just skewed symbol frequencies; it's *repetitive*. HTML repeats the same tag-and-attribute boilerplate,
JSON repeats key names, English repeats `the`. **LZ77** (1977) replaces a repeated run with a
back-reference: *"copy 14 bytes from 312 bytes ago."* The "dictionary" is simply the recent
output — a sliding window over what you've already seen. Long repeats collapse to a few bytes.

## DEFLATE = LZ77 + Huffman, and it's everywhere

<Callout variant="insight" title="The combination that won">
Run LZ77 to turn repeats into back-references, then Huffman-code the result (both the literals and
the references) to squeeze the remaining redundancy. That's **DEFLATE** (born in PKZIP ~1993, standardized as RFC 1951 in 1996) — and
it's the engine inside **gzip**, **zlib**, **PNG**, **ZIP**, and HTTP's `Content-Encoding: gzip`.
One algorithm, born in PKZIP, quietly running under a third of the web. **Brotli** (Google, 2015)
sharpens both stages and ships a **built-in dictionary of ~13,000 common web strings**, so even a
*first*, never-before-seen response compresses against known HTML/CSS/JS boilerplate — it routinely
beats gzip by 15–25% on web assets. **Zstandard** (Facebook, RFC 8878) hits gzip-class ratios at
multiples of the speed and adds *trainable* dictionaries for many-small-payloads workloads —
increasingly the default where you control both ends.
</Callout>

## Lossy is a different game

The above is **lossless** — decompress to the exact input, mandatory for code and text. Media
plays differently: **JPEG**, **H.264/H.265**, **AAC** are **lossy** — they transform data into
the frequency domain (the discrete cosine transform) and *discard* detail human perception barely
registers, then losslessly pack what remains. Unrecoverable by design, and the only way to get
100:1 on a photo. The choice — lossless for meaning, lossy for perception — is itself a
Presentation-layer decision.

## The trap: compression leaks secrets

<Callout variant="security" title="CRIME, BREACH, and why TLS dropped compression">
Compression's whole job is *measuring repetition* — so the compressed **size** reveals whether
attacker-influenced input matched a secret in the same stream. The **CRIME** (2012) and **BREACH**
(2013) attacks exploited exactly this: inject guesses next to a session token inside one
compressed-then-encrypted response, and watch the ciphertext *shrink* when a guess prefix matches —
recovering the secret byte by byte, straight through the encryption. Encryption hides content but
**not length**. The fixes were structural: TLS 1.3 **removed compression entirely**, and the
standing rule is that **secrets and attacker-controlled input must never share a compression
context**. A beautiful, counterintuitive lesson — that making data smaller can make it leak.
</Callout>
