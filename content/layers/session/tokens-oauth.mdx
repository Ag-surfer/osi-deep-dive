## Keeping a user logged in over a stateless protocol

[HTTP is stateless](../../application/http/) — it remembers nothing between requests — yet a web app must
keep you logged in across many of them. That is a **session** problem, solved two ways:

- **Server-side sessions** — the server stores the session state and gives the browser an opaque
  **session-ID cookie**. Stateful (needs a shared store across servers), but trivially **revocable**:
  delete the server record and the session is dead.
- **Tokens** — the server hands the client a **self-contained, signed token** (a **JWT**) carrying the
  identity and claims. The server verifies the signature on each request with **no lookup** — stateless
  and scalable — but that means it can't easily **revoke** a token before it expires.

<Callout variant="note" title="JWT and bearer tokens, briefly">
A **JWT** is `header.payload.signature`: a base64url payload of claims (who, what scopes, expiry) plus a
signature the server checks. Because verification needs no database hit, JWTs scale horizontally — at the
cost of revocation (you can't un-issue one), which is why access tokens are kept **short-lived** and
paired with a **refresh token**. Most of these are **bearer tokens**: *whoever holds the token can use
it*, no further proof — so protecting them (HTTPS, secure storage) is the whole ballgame.
</Callout>

## OAuth 2.0: delegated authorization

OAuth answers a different question: *"How do I let app X access my data on service Y **without giving X
my Y password**?"* — "Let this app see your Google Calendar." It defines four roles: the **resource
owner** (you), the **client** (the app), the **authorization server** (issues tokens), and the
**resource server** (holds the data). The client never sees your password; it receives a scoped,
revocable **access token** instead.

<Callout variant="insight" title="Why the indirection — code, then token">
Handing the token straight to the browser would expose it in the URL/history. The **authorization
code** is a short-lived, single-use stand-in: it's delivered via redirect, then exchanged **server-to-
server** (or with **PKCE** for public clients that have no secret) for the real token over a back
channel. **PKCE** (<RFCRef n={7636} />) stops an attacker who intercepts the code from redeeming it,
because they lack the original secret verifier. It's now recommended for *all* clients, not just mobile.
</Callout>

## OIDC and SAML: from authorization to login

OAuth grants *access*; it doesn't, by itself, tell the app **who you are**. **OpenID Connect (OIDC)**
adds an identity layer on top: an **ID token** (a JWT of identity claims) so "Log in with Google/Apple"
works. In the enterprise, **SAML** plays the same browser-SSO role with XML assertions — older and
heavier, but still everywhere in corporate single sign-on.

## Failure modes & troubleshooting

<Callout variant="warning" title="Symptoms and the cause behind them">
- **Stolen token = full access** — bearer tokens have no holder check; leaked via XSS, logs, or an
  insecure store. Keep them short-lived, HTTPS-only, and out of `localStorage` where XSS can read them.
- **Logout doesn't actually revoke** — a stateless JWT stays valid until it expires; use short access-
  token lifetimes plus a refresh-token revocation list, or server-side sessions where instant revocation
  matters.
- **`redirect_uri` mismatch / open redirect** — the auth server must match the redirect URI exactly;
  loose validation enables token theft.
- **Token expired mid-session** — the client must use the refresh token to get a new access token
  transparently, or the user gets logged out unexpectedly.
</Callout>
